# Terraform Security Best Practices — Production Hardening Guide

A comprehensive guide to securing Terraform workflows, state files, secrets, and CI/CD pipelines. Covers state protection, secrets management, provider security, policy enforcement, and supply chain integrity.

---

## 🔐 1. State File Security

The Terraform state file is the most sensitive artifact in your IaC pipeline. It contains **every resource attribute including passwords, private keys, and API tokens in plain text**.

### Never Commit State to Git
```bash
# ✅ MANDATORY: Add to .gitignore
echo "*.tfstate" >> .gitignore
echo "*.tfstate.backup" >> .gitignore
echo ".terraform/" >> .gitignore
echo "*.tfvars" >> .gitignore   # May contain secrets
echo "crash.log" >> .gitignore
```

### Use Remote Backend with Encryption
```hcl
# ✅ GCS backend with encryption and versioning
terraform {
  backend "gcs" {
    bucket = "my-terraform-state"
    prefix = "prod/terraform.tfstate"
  }
}
```

```bash
# Create the bucket with security hardening
gcloud storage buckets create gs://my-terraform-state \
    --location=us-east1 \
    --uniform-bucket-level-access \
    --public-access-prevention=enforced

# Enable versioning (allows state recovery)
gcloud storage buckets update gs://my-terraform-state --versioning

# Enable object retention (prevent accidental deletion)
gcloud storage buckets update gs://my-terraform-state \
    --retention-period=30d
```

### Restrict State Bucket Access
```bash
# ❌ BAD: Broad access to state bucket
gcloud projects add-iam-policy-binding my-project \
    --member="group:developers@company.com" \
    --role="roles/storage.admin"  # Can read ALL secrets in state!

# ✅ GOOD: Only CI/CD service account can access state
gcloud storage buckets add-iam-policy-binding gs://my-terraform-state \
    --member="serviceAccount:terraform-ci@my-project.iam.gserviceaccount.com" \
    --role="roles/storage.objectAdmin"
```

### Enable State Locking
```hcl
# GCS backend has built-in locking (no extra config needed)
# For S3 backend, always configure DynamoDB:
terraform {
  backend "s3" {
    bucket         = "my-terraform-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"   # Required for locking
    encrypt        = true
  }
}
```

## 🔑 2. Secrets Management

### Never Hardcode Secrets in Terraform Files
```hcl
# ❌ BAD: Password in HCL file (will be committed to Git AND stored in state)
resource "google_sql_database_instance" "db" {
  root_password = "SuperSecretPassword123"
}

# ❌ BAD: Even terraform.tfvars is dangerous if committed
# terraform.tfvars
db_password = "SuperSecretPassword123"
```

### Use Secret Manager or Environment Variables
```hcl
# ✅ GOOD: Read from GCP Secret Manager
data "google_secret_manager_secret_version" "db_password" {
  secret = "prod-db-password"
}

resource "google_sql_database_instance" "db" {
  root_password = data.google_secret_manager_secret_version.db_password.secret_data
}
```

```bash
# ✅ GOOD: Pass via environment variable (for CI/CD)
export TF_VAR_db_password="SuperSecretPassword123"
terraform apply  # Terraform reads TF_VAR_* automatically
```

### Mark Sensitive Variables
```hcl
# ✅ Prevents values from appearing in CLI output and logs
variable "db_password" {
  type      = string
  sensitive = true  # Terraform will mask this in plan/apply output
}

output "db_connection_string" {
  value     = "mysql://${var.db_user}:${var.db_password}@${google_sql_database_instance.db.private_ip_address}/app"
  sensitive = true  # Required if any input is sensitive
}
```

### Use `sensitive` Function for Computed Values
```hcl
# Mark a value as sensitive inline
locals {
  api_key = sensitive(data.google_secret_manager_secret_version.api_key.secret_data)
}
```

## 🏗️ 3. Provider and Module Security

### Pin Provider Versions
```hcl
# ❌ BAD: No version constraint (uses whatever is latest)
terraform {
  required_providers {
    google = {
      source = "hashicorp/google"
    }
  }
}

# ✅ GOOD: Pin to major version with pessimistic constraint
terraform {
  required_version = ">= 1.5.0"
  required_providers {
    google = {
      source  = "hashicorp/google"
      version = "~> 6.0"  # Any 6.x, but not 7.0
    }
  }
}
```

### Use Lock Files
```bash
# ✅ Always commit the lock file (ensures reproducible builds)
# .terraform.lock.hcl is auto-generated by terraform init
git add .terraform.lock.hcl

# Update lock file when upgrading providers
terraform init -upgrade
```

### Pin Module Versions
```hcl
# ❌ BAD: No version — could pull breaking changes
module "vpc" {
  source = "terraform-google-modules/network/google"
}

# ✅ GOOD: Pinned version
module "vpc" {
  source  = "terraform-google-modules/network/google"
  version = "~> 9.0"
}

# ✅ BEST: Pin to exact version for critical infrastructure
module "gke" {
  source  = "terraform-google-modules/kubernetes-engine/google"
  version = "35.0.0"
}
```

### Use Private Module Registries for Internal Modules
```hcl
# ✅ Host internal modules in a private registry or Git
module "internal_vpc" {
  source = "git::https://github.com/my-org/terraform-modules.git//vpc?ref=v2.1.0"
}
```

## 🚦 4. Policy Enforcement (Guardrails)

### Use `checkov` for Static Analysis
```bash
# Scan Terraform files for security misconfigurations
pip install checkov
checkov -d .

# Common findings:
# - GCS bucket without versioning
# - Firewall rule with 0.0.0.0/0 source range
# - Cloud SQL without backup configuration
# - GKE cluster without network policy
```

### Use `tfsec` for Security Scanning
```bash
# Install and run tfsec
brew install tfsec
tfsec .

# Example output:
# CRITICAL: google_compute_firewall.allow_all has source_ranges 0.0.0.0/0
# HIGH: google_storage_bucket.data has no versioning enabled
# HIGH: google_sql_database_instance.db has no backup_configuration
```

### Use OPA/Conftest for Custom Policy Rules
```bash
# Install conftest
brew install conftest
```

```rego
# policy/terraform.rego — Block public GCS buckets
package main

deny[msg] {
  resource := input.resource_changes[_]
  resource.type == "google_storage_bucket"
  not resource.change.after.uniform_bucket_level_access
  msg := sprintf("Bucket '%s' must enable uniform_bucket_level_access", [resource.change.after.name])
}

# Block overly permissive firewall rules
deny[msg] {
  resource := input.resource_changes[_]
  resource.type == "google_compute_firewall"
  some i
  resource.change.after.source_ranges[i] == "0.0.0.0/0"
  some j
  resource.change.after.allow[j].ports[_] != "80"
  resource.change.after.allow[j].ports[_] != "443"
  msg := sprintf("Firewall '%s' allows 0.0.0.0/0 on non-HTTP ports", [resource.change.after.name])
}
```

```bash
# Run policy checks against terraform plan output
terraform plan -out=plan.tfplan
terraform show -json plan.tfplan > plan.json
conftest test plan.json
```

### Use Terraform Sentinel (HashiCorp Enterprise)
```hcl
# Sentinel policy example — enforce tags on all resources
policy "require-labels" {
  enforcement_level = "hard-mandatory"
}
```

## 🔄 5. CI/CD Pipeline Security

### Use Separate Service Accounts per Environment
```bash
# ✅ Each environment gets its own SA with different permissions
gcloud iam service-accounts create terraform-dev \
    --display-name="Terraform Dev"
gcloud iam service-accounts create terraform-prod \
    --display-name="Terraform Prod"

# Dev SA: broader permissions for experimentation
# Prod SA: read-heavy, minimal write permissions
```

### Use Workload Identity Federation (No JSON Keys)
```bash
# ❌ BAD: Storing JSON key files in CI/CD secrets
gcloud iam service-accounts keys create key.json

# ✅ GOOD: Use OIDC federation for GitHub Actions
gcloud iam workload-identity-pools create github-pool \
    --location=global \
    --display-name="GitHub Pool"

gcloud iam workload-identity-pools providers create-oidc github-provider \
    --location=global \
    --workload-identity-pool=github-pool \
    --issuer-uri="https://token.actions.githubusercontent.com" \
    --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository"
```

### Require Plan Approval Before Apply
```yaml
# GitHub Actions workflow — plan on PR, apply on merge
name: Terraform
on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  plan:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform plan -no-color
        # Post plan output as PR comment for review

  apply:
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    environment: production  # Requires manual approval
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - run: terraform apply -auto-approve
```

## 🧹 6. Code Hygiene

### Use `terraform fmt` and `terraform validate`
```bash
# Format all files (idempotent)
terraform fmt -recursive

# Validate syntax and configuration
terraform validate

# Add to pre-commit hooks
# .pre-commit-config.yaml
repos:
- repo: https://github.com/antonbabenko/pre-commit-terraform
  hooks:
  - id: terraform_fmt
  - id: terraform_validate
  - id: terraform_tflint
  - id: terraform_checkov
```

### Use `moved` Blocks Instead of `state mv`
```hcl
# ✅ Terraform 1.1+: Declare renames in code (auditable, reviewable)
moved {
  from = google_storage_bucket.old_name
  to   = google_storage_bucket.new_name
}
```

### Use `prevent_destroy` Lifecycle for Critical Resources
```hcl
resource "google_sql_database_instance" "production_db" {
  name = "prod-db"
  # ...

  lifecycle {
    prevent_destroy = true  # Terraform will error if someone tries to destroy this
  }
}
```

## 🛡️ 7. Security Checklist

| Category | Best Practice | Priority |
|---|---|---|
| **State** | Never commit `.tfstate` to Git | 🔴 Critical |
| **State** | Use remote backend with encryption | 🔴 Critical |
| **State** | Restrict state bucket access to CI/CD SA only | 🔴 Critical |
| **State** | Enable versioning on state bucket | 🔴 Critical |
| **Secrets** | Never hardcode secrets in HCL or tfvars | 🔴 Critical |
| **Secrets** | Use Secret Manager or `TF_VAR_` env vars | 🔴 Critical |
| **Secrets** | Mark sensitive variables with `sensitive = true` | 🟡 High |
| **Providers** | Pin provider and module versions | 🔴 Critical |
| **Providers** | Commit `.terraform.lock.hcl` | 🟡 High |
| **Providers** | Use private registries for internal modules | 🟢 Medium |
| **Policy** | Run `checkov` or `tfsec` in CI pipeline | 🟡 High |
| **Policy** | Use OPA/Conftest for custom policies | 🟢 Medium |
| **CI/CD** | Use Workload Identity Federation (no JSON keys) | 🔴 Critical |
| **CI/CD** | Require plan review before apply | 🔴 Critical |
| **CI/CD** | Separate SA per environment | 🟡 High |
| **Code** | Use `prevent_destroy` on critical resources | 🟡 High |
| **Code** | Use `moved` blocks instead of `state mv` | 🟢 Medium |
| **Code** | Run `terraform fmt` in pre-commit hooks | 🟢 Medium |
